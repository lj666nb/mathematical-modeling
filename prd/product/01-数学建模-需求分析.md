## 一、学科痛点深度分析

### 1.1 痛点全景图

数学建模是连接数学理论与现实问题的桥梁学科，但目前的教学和学习过程存在 **7 大核心痛点**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                     数学建模教学的 7 大痛点                            │
├───────────────────┬───────────────────┬─────────────────────────────┤
│   学习层面          │   实践层面          │   产出层面                   │
├───────────────────┼───────────────────┼─────────────────────────────┤
│ ① 入门门槛高       │ ③ 理论与实践脱节    │ ⑤ 论文撰写困难               │
│ ② 知识体系碎片化    │ ④ 编程能力不足      │ ⑥ 反馈周期长                 │
│                   │                   │ ⑦ 论文质量不可控              │
└───────────────────┴───────────────────┴─────────────────────────────┘
```

### 1.2 痛点逐一分析

#### 痛点 ①：入门门槛高——"不知道从哪开始"

| 维度 | 分析 |
|------|------|
| **问题表现** | 数学建模涉及优化、统计、微分方程、图论、随机过程等多个数学分支，学生面对庞杂的知识体系无从下手 |
| **典型场景** | 大一/大二学生第一次接触数学建模："该先学什么？""这么多模型我该选哪个？" |
| **根因分析** | ① 数学建模是交叉学科，缺少统一的知识结构框架；② 高校课程偏理论推导，少实践引导；③ 缺乏针对不同基础的个性化学习路径 |
| **量化影响** | 以 CUMCM 为例，每年约 15 万参赛者中，首次参赛者占比超 60%，其中约 40% 在备赛初期因"不知道该学什么"而放弃系统训练 |
| **AI 可介入点** | 基于用户基础评估提供个性化学习路径；通过对话式交互降低认知负荷；用决策树辅助模型选择 |

#### 痛点 ②：知识体系碎片化——"学完了还是串不起来"

| 维度 | 分析 |
|------|------|
| **问题表现** | 学生分别学了线性规划、回归分析、聚类算法，但遇到实际问题时无法判断"这道题该用什么方法" |
| **典型场景** | 期末考试前学生问："老师讲了 10 种模型，但考试时我怎么知道选哪个？" |
| **根因分析** | ① 课程按方法分章节教学，缺少跨方法的横向对比；② 缺少"从问题出发选择方法"的逆向训练；③ 案例教学不足，学生没见过足够多的"真实问题→模型匹配"样本 |
| **AI 可介入点** | 8 大模型分类知识库 + 决策树规则引擎；对话中展示相同问题用不同方法解的效果对比；案例库标注"问题特征→模型选择"的映射关系 |

#### 痛点 ③：理论与实践脱节——"课上听懂了，做题就懵"

| 维度 | 分析 |
|------|------|
| **问题表现** | 课堂侧重定理推导和方法讲解，学生缺少完整的"问题→建模→求解→分析→论文"全流程训练 |
| **典型场景** | 学生听完优化的课，能复述单纯形法原理，但给他一个"工厂生产计划"问题时，无法完成"问题建模→数学表达→求解"的转化 |
| **根因分析** | ① 理论课时与实验课时比例失衡（多数高校为 3:1 甚至 4:1）；② 缺少端到端的项目式训练；③ 教师批改负担重，难以给每个学生详细的过程性反馈 |
| **AI 可介入点** | S0-S8 全流程工作流覆盖"读题→建模→代码→论文"；AI 在每个阶段提供即时引导而非直接给答案；竞赛模式提供完整的端到端体验 |

#### 痛点 ④：编程能力不足——"公式会写，代码不会写"

| 维度 | 分析 |
|------|------|
| **问题表现** | 非 CS 专业学生（数学、物理、工程）能将模型写成数学表达式，但无法用 Python/MATLAB 实现 |
| **典型场景** | 学生："我知道这是线性规划，约束条件也列好了，但 `scipy.optimize.linprog` 怎么用？""矩阵维度不匹配怎么办？" |
| **根因分析** | ① 非 CS 专业编程训练不足；② 数学软件（MATLAB/Lingo）需付费，Python 生态虽免费但学习曲线陡峭；③ 代码调试能力薄弱，遇到报错就卡住 |
| **量化影响** | 根据 CUMCM 参赛经验，约 50% 的队伍在竞赛第 1 天花费大量时间在代码调试上，而非建模思路 |
| **AI 可介入点** | 代码辅助 Agent 提供语法纠错 + 逻辑审查；引擎层自动生成代码脚手架（S5）；在线代码编辑器 + 沙箱一键运行 |

#### 痛点 ⑤：论文撰写困难——"模型跑出来了，论文写不出来"

| 维度 | 分析 |
|------|------|
| **问题表现** | 学生缺乏科技论文写作经验，不知道摘要怎么写、结果怎么展示、分析怎么展开 |
| **典型场景** | 竞赛队伍到第三天晚上还在赶论文："摘要应该写什么？""模型优缺点怎么表述？""参考文献格式不对？" |
| **根因分析** | ① 本科阶段科技论文写作训练普遍缺失；② 竞赛论文有特定规范（20-25 页、标题层级、公式编号等），学生不熟悉；③ 写作和建模是不同维度的能力，团队中可能无人擅长 |
| **AI 可介入点** | 论文评审 Agent：结构评审 + 公式审查 + 改进建议；引擎层 S7：自动 outline → Agent 全局写作 → 格式门禁；范文库提供历年优秀论文参考 |

#### 痛点 ⑥：反馈周期长——"交了论文要等一周才知道好不好"

| 维度 | 分析 |
|------|------|
| **问题表现** | 传统教学模式下，学生提交论文 → 教师批改 → 返回评语，周期通常 3-7 天，且评语多为结论性（打分+简短评语），缺少过程性指导 |
| **典型场景** | 学生交作业："老师我这道题做对了吗？"→ 3 天后回复："基本正确，但灵敏度分析不够充分。"→ 学生已进入下一章内容 |
| **根因分析** | ① 师生比低（1 个教师面对 50-100 名学生）；② 数学建模作业评阅工作量大（既要看建模思路又要看代码又要看论文）；③ 缺乏自动化的过程性评价工具 |
| **AI 可介入点** | 教学 Agent 7×24 即时答疑；论文评审 Agent 秒级给出结构化评语；证据门禁 S6 自动检查建模结果是否具备真实证据支撑 |

#### 痛点 ⑦：论文质量不可控——"AI 辅助容易'看起来对但实际没跑过代码'"

| 维度 | 分析 |
|------|------|
| **问题表现** | 学生用通用 AI 工具（ChatGPT 等）辅助论文写作时，AI 容易"编造"数据、图表、公式——看起来合理但未经真实计算验证 |
| **典型场景** | 学生在通用 AI 中问"帮我写一段层次分析法的分析"，AI 生成了"CR=0.032<0.1"和精美表格——但数据是编的 |
| **根因分析** | ① 通用 AI 不执行代码，无法验证结果真实性；② 学生缺乏辨别 AI 编造内容的能力；③ 没有系统化的质量门禁机制 |
| **AI 可介入点** | **证据门禁（S6）**：每问检查是否具备真实计算/图表/表格证据；**格式门禁（S7）**：检查字数、标题层级、引用规范；**多 Agent 协同**：建模 Agent 生成代码 → 沙箱执行 → 审核 Agent 验证结果 → 写入 Agent 引用真实数据 |

### 1.3 痛点与功能映射

| 痛点 | 对应功能模块 | 对应 Feature ID |
|------|------------|----------------|
| ① 入门门槛高 | 学习中心（知识库、学习路径） | LRN-001 ~ LRN-004 |
| ② 知识体系碎片化 | 学习中心 + 建模引导 Agent | LRN-001, AGT-002 |
| ③ 理论与实践脱节 | 建模工作台 + 竞赛训练 | WRK-001 ~ WRK-007, CMP-001 ~ CMP-014 |
| ④ 编程能力不足 | 建模工作台 + 代码辅助 Agent | WRK-001 ~ WRK-003, AGT-003 |
| ⑤ 论文撰写困难 | 论文评审 Agent + 竞赛训练 S7 | AGT-004, CMP-008 ~ CMP-009 |
| ⑥ 反馈周期长 | AI 智能体对话（全部 Agent） | AGT-001 ~ AGT-008 |
| ⑦ 论文质量不可控 | 竞赛训练（证据门禁+格式门禁） | CMP-007, CMP-009 |

### 2.2 用户故事（User Stories）

#### 用户画像 ①：数学建模竞赛参赛者（P0 核心用户）

**故事 1.1：备赛训练**
> **作为** 正在准备 CUMCM 的大二工科生，  
> **我想** 系统性地训练建模能力，了解每类模型的适用场景和代码实现，  
> **以便** 在竞赛时能快速判断"这道题该用什么方法"并有现成的代码模板可用。

**故事 1.2：竞赛实战（学习模式辅助）**
> **作为** 正在参加 72 小时竞赛的队员，  
> **我想** 在遇到代码报错或方法不确定时立刻得到 AI 的精准帮助，  
> **以便** 不因卡在技术细节而浪费宝贵的竞赛时间。

**故事 1.3：竞赛实战（竞赛模式—引擎驱动）**
> **作为** 竞赛队伍的主力建模手，  
> **我想** 将赛题上传后，系统自动帮我完成"审题→模型路线→数据处理→建模代码→证据检查→论文成稿"的全流程，  
> **以便** 在 72 小时内高效产出一篇结构完整、有真实数据支撑的论文。

**故事 1.4：论文收尾**
> **作为** 竞赛队伍中负责写论文的队员，  
> **我想** AI 帮我检查论文摘要是否到位、结构是否完整、格式是否规范，  
> **以便** 在提交截止前确保论文符合竞赛评分标准。

#### 用户画像 ②：理工科本科生（P0 核心用户）

**故事 2.1：课堂同步学习**
> **作为** 正在上数学建模课的大三数学系学生，  
> **我想** 在课后向 AI 追问课堂没听懂的概念（如"层次分析法的一致性检验为什么要 CR<0.1？"），  
> **以便** 真正理解而非死记硬背。

**故事 2.2：作业辅助**
> **作为** 被布置了一道优化建模作业的学生，  
> **我想** AI 引导我"先想清楚决策变量是什么、目标函数是什么、约束条件有哪些"，而不是直接给我答案，  
> **以便** 培养独立建模的思维习惯。

**故事 2.3：代码实现**
> **作为** 数学系学生（Python 水平一般），  
> **我想** AI 帮我解释建模代码的每一部分在做什么并指出我的错误，  
> **以便** 我能理解代码逻辑、下次能自己写。

**故事 2.4：期末复习**
> **作为** 临近期末考试的学生，  
> **我想** 快速回顾所有建模方法的核心思想和适用场景对比，  
> **以便** 考试时能准确判断题目类型并选择正确方法。

#### 用户画像 ③：数学建模课程教师（P1）

**故事 3.1：教学资源管理**
> **作为** 教授数学建模课程的大学教师，  
> **我想** 有一个平台能组织教学案例、经典模型、优秀论文等资源，  
> **以便** 减少备课负担、提升课堂质量。

**故事 3.2：学生进度追踪**
> **作为** 希望了解学生真实水平的教师，  
> **我想** 看到每个学生的 AI 对话摘要、练习记录和常见错误类型，  
> **以便** 针对性地调整教学内容和方法。

**故事 3.3：减轻批改负担**
> **作为** 面对 60 人班级作业批改任务的教师，  
> **我想** AI 先做出初步的论文评审（结构、逻辑、格式），标注出问题段落，  
> **以便** 我把精力集中在深度评价而非机械检查上。

#### 用户画像 ④：数学建模爱好者/自学者（P2）

**故事 4.1：找到学习起点**
> **作为** 对数学建模感兴趣但完全零基础的自学者，  
> **我想** 先做一个"水平测试"了解自己该从哪里开始，  
> **以便** 不会一来就看到看不懂的内容而放弃。

**故事 4.2：碎片化学习**
> **作为** 利用业余时间学习建模的上班族，  
> **我想** 每次 15-30 分钟学完一个知识点并有相应练习，  
> **以便** 在碎片时间里持续进步。

## 三、竞品分析

### 3.1 竞品全景图

```
                        数学建模
                       辅助工具
                          │
        ┌─────────────────┼─────────────────┐
        │                 │                 │
   通用 AI 助手       专业建模软件       在线教育平台
```

### 3.2 竞品详细对比

| 维度 | 通用 AI 助手（ChatGPT/Claude/Kimi等） | 专业建模软件（MATLAB/LINGO/Mathematica） | MathModel Skill 引擎 | **本产品（双层平台）** |
|------|:---:|:---:|:---:|:---:|
| **能力范围** | 对话+推理 | 数值计算+可视化 | 完整论文工作流 | 教学+论文工作流 |
| **交互方式** | 纯文本对话 | GUI + 编程 | CLI（Claude Code Agent） | Web UI + 对话 + CLI |
| **针对数学建模的优化** | ❌ 通用能力，无专门优化 | ⚠️ 工具强但无教学引导 | ✅ 完全针对数学建模论文生产 | ✅ 同时覆盖教学和论文生产 |
| **模型选择引导** | ⚠️ 能回答但无结构化决策 | ❌ 不提供 | ✅ S2 结构化模型路线 | ✅ 教学 Agent + S2 引擎 |
| **代码生成与执行** | ⚠️ 能生成代码但不执行 | ✅ 专业级执行 | ✅ 生成脚手架 + 约定契约 | ✅ 代码辅助 Agent + 沙箱执行 |
| **论文写作** | ❌ 容易编造数据 | ❌ 不提供 | ✅ S7 正式成稿+格式门禁 | ✅ S7 引擎+论文评审 Agent |
| **质量门禁** | ❌ 无 | ❌ 无 | ✅ 证据门禁+格式门禁 | ✅ 双门禁（继承引擎） |
| **教学引导** | ⚠️ 被动回答，不主动引导 | ❌ 无 | ❌ 只执行不教学 | ✅ 6大教学原则+引导式教学 |
| **用户画像适配** | ❌ 不区分用户 | ❌ 不适用 | ❌ 单一场景（竞赛） | ✅ 4 类画像×差异策略 |
| **开放性** | 封闭（厂商模型） | 封闭（商业许可） | 开放（用户自带Key） | 开放（用户自带Key） |
| **部署门槛** | 零（SaaS） | 高（商业软件） | 中（需Claude Code CLI） | 低（Docker一键部署） |
| **价格** | 免费/订阅 | 昂贵（$1000+/年） | 开源免费 | 开源免费 |

### 3.3 MathModel Skill 引擎的定位分析

MathModel Skill 是本产品的**论文引擎层**，我们需要清晰理解它的定位：

| 维度 | MathModel Skill 的优势 | MathModel Skill 的局限（本产品要补足） |
|------|----------------------|-------------------------------------|
| **用户体验** | CLI/Agent 原生态，适合技术用户 | 无 Web UI，普通学生难以使用 → **教学平台层提供 Web UI** |
| **教学模式** | 纯执行模式，"做"但不"教" | 不会解释"为什么选这个模型"→ **教学 Agent 补齐教学能力** |
| **交互粒度** | 粗粒度（一次跑一个 Skill） | 无互动式的分步引导 → **竞赛模式 Web UI 分步展示** |
| **错误恢复** | 门禁检查后可手动重跑 | 用户不知道如何修正 → **桥接层翻译门禁报告为可操作建议** |
| **上下文保持** | 进程内全流程记忆 | 无跨会话持久化 → **后端 Redis 会话缓存 + 数据库持久化** |

### 3.4 差异化优势总结

```
┌──────────────────────────────────────────────────────────────────┐
│                      本产品的差异化优势                            │
│                                                                  │
│  ① "教+做"一体化                                                  │
│     竞品要么只教（MOOC）要么只做（MATLAB），本产品教学层+引擎层      │
│     配合，学完就能用                                              │
│                                                                  │
│  ② 质量门禁系统                                                    │
│     业界唯一具备"证据门禁+格式门禁"的数学建模辅助工具，             │
│     确保产出结果真实可靠                                          │
│                                                                  │
│  ③ Agent-native 工作流 Web 化                                    │
│     首次将 Claude Code Agent Skill 工作流 Web 化，                │
│     普通学生也能享受 Agent 自动化论文生产                          │
│                                                                  │
│  ④ 引导式教学而非直接给答案                                        │
│     6 大教学原则确保 AI 老师"授人以渔"而非"授人以鱼"               │
│                                                                  │
│  ⑤ 用户自带 LLM 密钥                                             │
│     开放的 LLM 生态，不绑定特定厂商，支持 OpenAI/DeepSeek/         │
│     通义千问/GLM 等多厂商                                         │
│                                                                  │
│  ⑥ 全流程证据链                                                   │
│     问题分析→模型路线→数据→代码→结果→图表→论文，                   │
│     每一步都有 JSON 契约追溯，避免"拍脑袋"写论文                   │
└──────────────────────────────────────────────────────────────────┘
```

### 3.5 竞争风险与对策

| 风险 | 严重程度 | 应对策略 |
|------|:---:|------|
| 通用 AI 持续进化（ChatGPT 等提高数学推理能力） | 中 | 持续强化领域专用性——通用 AI 不会做门禁检查、不走 Agent 工作流、不区分用户画像 |
| MathModel Skill 自身发展出 Web UI | 低 | 本产品与其是"互补"关系而非竞争，且引擎层可替换 |
| 高校自研类似平台 | 低 | 高校自研项目通常缺乏持续维护能力，且不会整合 Agent 工作流 |
| LLM API 费用波动 | 中 | 用户自带密钥 + 多厂商切换，不承担 API 成本风险 |

**MVP 完成后用户能做什么**：

## 五、需求边界与约束

### 5.1 明确包含（In Scope）

- ✅ 数学建模 8 大模型分类的结构化知识内容
- ✅ AI 对话式教学（4 个教学 Agent）
- ✅ 在线 Python 代码编辑 + Docker 沙箱执行
- ✅ MathModel Skill 引擎的 Web 化封装（S0-S8）
- ✅ 用户自带 LLM 密钥的多厂商适配
- ✅ Docker Compose 一键部署

### 5.2 明确不包含（Out of Scope）

- ❌ 数学公式的符号计算（如符号积分、符号微分）——这是 CAS（计算机代数系统）的范畴
- ❌ 自研 LLM 模型——依赖外部 API
- ❌ 实时多人协作编辑（如 Google Docs 式协作）——复杂度高，非 P0
- ❌ 移动端 App——三期考虑 PWA 适配
- ❌ 自动评分（给论文打具体分数）——AI 评分缺少标准，仅做定性评审
- ❌ 竞赛题目答案的直接生成——遵守学术诚信原则

### 5.3 技术约束

| 约束 | 说明 | 影响 |
|------|------|------|
| Claude Code Agent 依赖 | 竞赛模式 S0-S8 需要 Claude Code CLI 运行时 | 部署时必须安装 Claude Code CLI 或等效 API |
| 用户自有 LLM 密钥 | 系统不自带 LLM 密钥，用户需自己提供 | 首次使用有配置门槛，需提供详细的配置引导 |
| 代码沙箱安全 | Docker 沙箱必须正确配置以防止逃逸 | 沙箱容器需禁用网络、限制 CPU/内存/磁盘 |
| MathModel Skill 版本锁定 | 引擎层依赖 MathModel Skill v2.0.0 | 升级需同步验证工作流桥接层的兼容性 |
| Windows/Linux 兼容 | 部分 Python 脚本在 Windows 下路径处理可能不同 | 优先 Linux（Docker），Windows 需额外测试 |

### 5.4 学术诚信约束

| 原则 | 具体措施 |
|------|---------|
| **引导而非代做** | 学习模式 Agent 遵循"引导式教学"原则，不直接输出完整答案 |
| **透明化** | 竞赛模式生成的论文标注 AI 辅助工具的使用，建议用户标注 |
| **可验证性** | 所有建模结果必须附带代码和真实数据，可通过证据门禁追溯 |
| **用户自行判断** | 最终论文内容由用户自行审核和负责，AI 是辅助工具而非决策者 |
